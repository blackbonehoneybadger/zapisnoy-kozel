# DOFFA Crazy 8 — Аудит игры (Этап 1)

> Статус документа: **аудит текущего состояния репозитория** на момент начала работ
> по превращению игры в полноценный продукт экосистемы DOFFA.
> Механика в этом этапе **не менялась** — только исследование и фиксация фактов.

> `Telegram Tap → Cups → вход в матч → победа → «Забрать награду» → серверная проверка → DOFFA на Solana-кошелёк`

## 2. Архитектура сервера

**Стек:** Node.js + WebSocket (`ws`), TypeScript, `@solana/web3.js`. Разворачивается на Railway. Каталог `server/src/`:

| Файл | Назначение |
|---|---|
| `index.ts` | WebSocket-сервер, маршрутизация сообщений, рассылка, выплата банка (`handlePayout`), авто-ходы ботов (`pumpBots`) |
| `lobby.ts` | столы и рассадка **в памяти** (`Map`), старт партии, проверка оплаты, ходы, определение победителя через состояние движка |
| `match.ts` | создание серверной партии из движка клиента, **редакция чужих рук** (`redactFor`) |
| `accounts.ts` | аккаунты по адресу кошелька, HMAC-токены сессии, хранение в **JSON-файле** `server/data/accounts.json` |
| `solana.ts` | горячий кошелёк сервера, `sendSol` (native SOL), `verifySignature`, `verifyPayment` |
| `protocol.ts` | типы сообщений WS |

**Важно:** всё состояние столов/матчей — **в памяти процесса**. Аккаунты — **JSON-файл**. Никакой БД нет. При рестарте сервера все активные столы и матчи теряются.

**WS-обработчики (`index.ts`):** `auth:nonce`, `auth:verify`, `auth`, `profile:setName`, `lobby:subscribe/unsubscribe`, `table:create/join/leave/start`, `game:move/next`, `invite:*`, `friend:*`, `wallet:register`, `wallet:pay`.
**Нет обработчиков `reward:*` / `cups:*` / `match:*` — серверной экономики Cups/DOFFA не существует.**

## 3. Карточный движок (Crazy 8)

**Файлы:** `src/game/` — `engine.ts`, `rules.ts`, `deck.ts`, `types.ts`, `bots.ts`, `scoring.ts`, `labels.ts`.

- Чистые функции: `createInitialState`, `applyMove`, `startNextRound`, `getCurrentPlayer`, `canPlayCard`, `hasPlayableCard`.
- Спец-карты: шестёрки, семёрки, восьмёрки, девятки, дамы (смена масти), короли, тузы (пропуск).
- Боты: `decideBotMove(state, difficulty)`.
- **Ключевой факт:** движок **переиспользуется и на клиенте, и на сервере** (`server/src/match.ts`, `server/src/lobby.ts` импортируют из `../../src/game/*`). Это делает возможной авторитетную серверную партию **без переписывания правил** — большой плюс.

**Оценка:** зрелый, детерминированный, переиспользуемый. **Не переписывать.** Пригоден как основа для серверного `MatchResult`.

## 4. Онлайн-режим

**Поток:** `auth` (подпись кошелька) → `lobby:subscribe` → `table:create/join` → (опц. `wallet:pay` при ставке) → `table:start` → `game:move` → `gameOver`.

- Сервер держит партию (`table.game`), **валидирует каждый ход** (`applyUserMove`: чей ход, допустимость), водит ботов (`botStep`/`pumpBots`).
- Клиенту рассылается **редактированное** состояние (`redactFor`) — чужие руки скрыты, видно только число карт. Хорошая защита от читов «вижу чужие карты».
- Победитель определяется сервером из состояния движка.

**Оценка:** онлайн-режим **действительно авторитетен по ходам и результату**. Это фундамент, на который можно повесить серверные Cups/DOFFA и MatchResult. Слабое место — состояние только в памяти (рестарт = потеря матчей) и отсутствие записи результата/награды.

## 5. Solana-интеграция

**Клиент (`src/solana/`):**
- Подключение внешнего кошелька, `signMessage` (вход), `sendBet` (перевод ставки на кошелёк сервера) — всё подписывает пользователь.
- Сеть из `VITE_SOLANA_NETWORK` (по умолчанию `mainnet-beta`).

**Сервер (`server/src/solana.ts`):**
- Горячий кошелёк из `SOLANA_PRIVATE_KEY` (**обязателен на mainnet**, иначе throw — правильно).
- `sendSol(to, lamports)` — **native SOL** через `SystemProgram.transfer`; идемпотентен при таймауте RPC (проверяет `getSignatureStatuses` перед признанием неудачи).
- `verifyPayment` — проверяет, что ставка реально пришла на кошелёк сервера (pre/post balances).
- `verifySignature` — ed25519, доказательство владения кошельком.
- Комиссия площадки `PLATFORM_FEE` (по умолчанию 5%).

**Критично для целевой модели:** сервер умеет только **native SOL**. **SPL-токен DOFFA** (`mint 57aAfCuXx7uuc8g8P9kTxR65TKQtZsFDJeKhdD5xu6uo`) **нигде не отправляется** — нет `@solana/spl-token`, нет associated token accounts, нет Reward Vault. Реальная выплата DOFFA потребует **новой серверной инфраструктуры**.

## 6. Текущая механика Cups

**Файл:** `src/store/rewardsStore.ts` (только клиент).
- `CUPS_PER_GAME = 10`, `CUPS_PER_WIN = 25`.
- `awardGame(won)` начисляет Cups **в localStorage** после любой локальной партии.
- Cups **не** используются как стоимость входа в матч (в коде входной платы Cups нет — только SOL-ставки).
- Баланс живёт **только в localStorage**, сервер о Cups не знает.

**Проблема:** клиент сам себе начисляет Cups; сброс localStorage/правка = произвольный баланс. Cups сейчас — чисто визуальные очки.

---

## 7. Текущая механика DOFFA

**Файл:** `src/store/rewardsStore.ts` (только клиент).
- `DOFFA_PER_WIN = 10`.
- `awardGame(won)` при победе начисляет **10 DOFFA в localStorage** — **включая офлайн-победу над ботами** (вызов в `src/store/gameStore.ts:100`, `commit()` на `gameOver`).
- `claim(wallet)` просто обнуляет `doffa`, увеличивает `doffaClaimed`, пишет запись истории — **без сервера, без транзакции, без токена**.

**Проблема (критическая):**
1. Клиент решает размер награды и сам её начисляет.
2. Офлайн-игра выдаёт «реальную» DOFFA.
3. Claim — фикция: никакой SPL-токен не отправляется. `doffaClaimed` — просто число в localStorage.

Это ровно то, что требуется устранить на этапах 2–5.

## 13. Старые SOL-ставки

**Где:** лобби (`src/screens/OnlineScreen.tsx` — создание стола со ставкой, оплата `sendBet`), сервер (`lobby.ts` `betLamports/potLamports`, `index.ts` `wallet:pay` + `handlePayout`).
- Реально переводят **native SOL**: игроки платят банк на кошелёк сервера, победитель получает банк минус комиссия.
- Уже есть защита: ставка только когда все места заняты людьми и все оплатили; идемпотентная выплата; фиксация банка при старте.

**Решение по целевой модели:** SOL **не должен быть основной валютой**. Нужно **убрать из главного пользовательского пути и спрятать за feature flag**, но **не удалять без аудита** — код завязан на реальный горячий кошелёк и выплаты. Сохранить отдельно до решения о полном удалении.

## 14. Потенциальные проблемы безопасности

1. **🔴 Клиентское начисление настоящей DOFFA** (`rewardsStore.awardGame`, вызывается офлайн) — игрок «зарабатывает» награду без сервера.
2. **🔴 Claim без сервера** — «вывод» существует только в localStorage; при подключении реальной выплаты в таком виде = раздача токенов кому угодно.
3. **🟠 Экономика только в localStorage** — Cups/DOFFA/история тривиально редактируются пользователем.
4. **🟠 Состояние матчей в памяти** — рестарт теряет матчи; нет записи результата → нельзя доказать победу задним числом.
5. **🟠 Аккаунты в JSON-файле** — гонки записи, нет транзакционности, не масштабируется.
6. **🟠 Нет античита за пределами валидации хода** — нет rate-limit, sequence number, контроля длительности матча, дневных лимитов.
7. **🟡 SOL mainnet по умолчанию** (`VITE_SOLANA_NETWORK`, серверный `SOLANA_NETWORK`) — реальные деньги в дефолте.
8. **🟢 Хорошо:** приватных ключей в клиенте нет; `AUTH_SECRET` и `SOLANA_PRIVATE_KEY` обязательны на mainnet; чужие руки скрыты; ходы валидируются; выплата идемпотентна.
